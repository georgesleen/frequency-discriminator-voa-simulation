# linewidth-freq-discriminator-simulation - project notes

## Architecture

Python port of a Lumerical CHARGE + MODE/FDE pipeline for a SOI PIN
modulator cross-section. Goal: numerically comparable outputs against a
collaborator's Lumerical run.

- `lumerical/` - Lumerical LSF reference scripts (`Charge_Simulation.lsf`,
  `Mode_Simulation.lsf`, `PN_Junction_Setup.lsf`), exported data
  (`lumerical_data.dat`), and `delay_time_calc.m`.
- `src/charge_sim.py` - DEVSIM drift-diffusion solve on an L-shaped silicon
  cross-section (gmsh mesh). Sweeps anode 0->4 V in 0.1 V steps; writes
  `src/carriers.npz` (electron + hole density on a 301x81 regular grid in
  SI units).
- `src/mode_sim.py` - femwell quasi-TE eigenmode solve at each voltage,
  Soref-Bennett perturbation from `carriers.npz`. Writes
  `src/modulator_neff_V.dat` and `output/neff_vs_V.png`.
- `src/visualize.py` - carrier density maps at four voltages -> `output/carrier_maps.png`.
- `src/compare_lumerical.py` - Python vs Lumerical overlay -> `output/compare_neff.png`.
- `src/params.py` - geometry / doping / sweep params, direct port of
  `lumerical/PN_Junction_Setup.lsf`. All values in SI; `charge_sim.py` converts to
  CGS at the DEVSIM boundary.
- `output/` - committed PNG plots regenerated by the pipeline.
- `flake.nix` - `buildFHSEnv` dev shell, see Workarounds.

## Running

```
nix run . -- -c "uv run src/charge_sim.py"
nix run . -- -c "uv run src/mode_sim.py"
```

(`nix develop` works too for interactive use.)

## Workarounds

- `flake.nix` uses `pkgs.buildFHSEnv` rather than `mkShell` -
  **Why:** DEVSIM's pip wheel is auditwheel'd against `manylinux2014` and
  does runtime `dlopen("libmkl_rt.so")` after Python is already running, so
  `nix-ld` (which only patches ELF interp at exec time) doesn't help.
  Faking FHS via `LD_LIBRARY_PATH` in a shellHook gets close but produces
  subtle symbol-resolution bugs; a real bubblewrap'd FHS at `/usr/lib`
  resolves cleanly.
- `flake.nix` uses `mkl` (Intel MKL, unfree, narrow `allowUnfreePredicate`)
  rather than `openblas` -
  **Why:** DEVSIM's bundled UMFPACK 5.1 was written against MKL's BLAS
  calling conventions. With OpenBLAS, UMFPACK calls `DGER` with a bad
  `LDA` (parameter 9) and segfaults during the Newton solve - even on
  DEVSIM's own `diode_1d.py` / `diode_2d.py` examples. INSTALL.md
  explicitly recommends MKL; supported platforms are AlmaLinux 8 / RHEL 8
  where MKL is the default. Symptoms with OpenBLAS:
  `** On entry to DGER parameter number 9 had an illegal value` then
  SIGSEGV (PY EXIT 139), occasionally masked to 0 by `nix run` framing.
- `src/charge_sim.py` mesh + doping expressions are in cm / cm^-3 (CGS),
  not SI -
  **Why:** DEVSIM's bundled silicon physics (`simple_physics.py`) is
  written in CGS; mixing SI mesh coords with CGS material parameters
  produces ~6-orders-of-magnitude unit mismatches that blow up the Newton
  solve. The `M2CM` / `M3_TO_CM3` / `CM3_TO_M3` constants do the
  conversion at the boundary; `carriers.npz` is converted back to SI for
  the mode solver.
- `src/charge_sim.py` writes gmsh MSH 2.2, not the gmsh-4.x default -
  **Why:** DEVSIM's MSH reader only handles legacy MSH 2.x.
  `Mesh.MshFileVersion = 2.2` set before `gmsh.write`.
- `pyproject.toml` pins `requires-python = ">=3.10,<3.13"` -
  **Why:** `vtk 9.3.1` (femwell transitive dep) has no Python 3.13 wheel.
- `src/charge_sim.py` floors `Donors` with `DONOR_FLOOR = 1 cm^-3` (`set_doping`) -
  **Why:** Klaassen mobility computes `Z_D ~ (Nref_D/Donors)^2`, and DEVSIM
  raises a divide-by-zero on `pow(Donors, -1)` wherever the n-implant is
  absent (`Donors = 0`). A 1 cm^-3 floor is numerically negligible against the
  1e15 p-epi and 4e20 implant levels. Acceptors need no floor (p-epi background
  already keeps them >=1e15).
- `src/charge_sim.py` adds surface recombination via `SurfaceMask * Us /
  NodeVolume^0.5` inside the bulk `Urecomb` expression (`create_recombination`)
  rather than through a contact equation -
  **Why:** DEVSIM's `contact_equation` `edge_model` only evaluates on 1D
  contact boundary elements (not interior bulk triangle edges), so contact-based
  surface recombination completely decouples boundary nodes from the interior
  domain and has zero effect on the rib core density. The bulk-node-model
  approach adds the surface term to the normal FVM equation at boundary nodes,
  where it does couple through standard interior edge fluxes. `physics.SRV_SI_SIO2`
  is an effective fit parameter (calibrated to ~8e18 cm^-3 core injection at
  V=1.5 V), not a physical Si/SiO2 SRV (which would be 10-10000 cm/s).
